gorchestrator

Phase 6 Implementation Plan — Polish & Ship

📅 July 8, 2026 📝 Status: Draft 🎯 Prerequisite: Phase 5 complete

Goal

A team can deploy gorchestrator to production, operate it from the docs alone, audit every decision after the fact, and trust it under sustained load at the target scale (hundreds of users).

Workstreams

A
Audit Completeness
  • Every state-changing action writes an audit entry — verified by test, not by convention.
  • Dashboard audit views: per-issue timeline, per-user activity, exportable (JSONL/CSV).
B
Metrics Dashboard
  • Token burn per project/agent/model over time; cycle time per phase; human intervention rate; failure/escalation rates.
  • Sourced from data already persisted (events.jsonl, runs, decisions, audit) — aggregation and display, not new collection.
C
Retention & Cleanup (closes §17 Q14)
  • Per-project retention: workspaces (largest artifacts) auto-deleted N days after terminal state; issue artifacts retained or archived — they are the audit substance.
  • Suggested defaults: delete workspaces after 30 days, archive issue artifacts after 180, never silently delete audit rows.
  • Storage-usage view; manual purge with confirmation + audit entry.
D
Documentation
  • Architecture doc generated from spec.md's living content — not a fork of it.
  • Admin guide (deployment, OIDC, project/git config, adapters, budgets, escalation); user guide (submitting, reading runs, adjudicating well — feedback quality matters); API reference.
E
Deployment & Operations
  • Single static binary release, embedded dashboard assets; Dockerfile + Compose (volumes for SQLite + storage root; container-runtime socket documented for run_test).
  • Backup/restore: SQLite snapshot + storage-root sync; restore drill documented and tested — filesystem-authoritative reconciliation (§9.4) should make partial restores survivable; verify that claim.
  • Health/readiness endpoints; structured logging; optional OTel export (already transitively present via ADK).
F
Hardening
  • CI: build, vet, race-detector tests, lint.
  • Fuzz tests on path containment and JSON-RPC framing — the two parsing surfaces exposed to untrusted-ish input.
  • Load smoke test at target scale: sustained parallel issues at the worker cap; SQLite contention, SSE fan-out, storage growth observed.
  • Versioning policy, CHANGELOG, release engineering.

Success Criteria

  • Fresh machine → deployed, configured, first issue through the pipeline using only the docs.
  • Every decision in a month of simulated use is reconstructible from audit + artifacts.
  • Load smoke test passes at target scale with no lock errors or lost events.
  • Backup → destroy → restore drill succeeds.
