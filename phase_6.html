Phase 6 Implementation Plan — Polish & Ship

📅 July 10, 2026 (revised) 📝 Status: Draft 🎯 Prerequisite: Phase 5 complete

Landed foundations — do not regress

Same contracts as Phase 5 (project registry, flavors, description + attachments, phase-scoped drawer, workspace.zip). Retention and docs treat these as first-class.

  • Retention covers issue.md, attachments/, and implementation/workspace/.
  • User docs cover title + description + attachments; multi-phase drawer; Workspace zip.
  • API reference documents body/description, multipart submit, workspace.zip, drawer phase param.

Goal

A team can deploy gorchestrator to production, operate it from the docs alone, audit every decision after the fact, and trust it under sustained load at the target scale (hundreds of users).

Workstreams

A
Audit Completeness
  • Every state-changing action writes an audit entry — verified by test, not by convention.
  • Dashboard audit views: per-issue timeline, per-user activity, exportable (JSONL/CSV).
B
Metrics Dashboard
  • Token burn per project/agent/model over time; cycle time per phase; human intervention rate; failure/escalation rates.
  • Sourced from data already persisted — aggregation and display, not new collection.
C
Retention & Cleanup (closes §17 Q14)
  • Per-project retention: workspaces auto-deleted N days after terminal state; issue artifacts (issue.md, attachments/, phase dirs) retained or archived.
  • Suggested defaults: delete workspaces after 30 days, archive issue artifacts after 180, never silently delete audit rows.
  • Storage-usage view; manual purge with confirmation + audit entry.
  • Do not define “issue artifacts” as phase dirs only and orphan issue.md / attachments/.
D
Documentation
  • Architecture doc generated from living spec.md.
  • Admin guide; user guide (submit with description/attachments; multi-phase drawer; adjudicating well); API reference.
  • README kept current.
E
Deployment & Operations
  • Single static binary release; Dockerfile + Compose; container-runtime socket documented for run_test.
  • Backup/restore drill tested (filesystem-authoritative reconciliation §9.4).
  • Health/readiness; structured logging; optional OTel.
F
Hardening
  • CI: build, vet, race-detector tests, lint.
  • Fuzz path containment and JSON-RPC framing; attachment path sanitization stays in regression scope.
  • Load smoke test at target scale.
  • Versioning policy, CHANGELOG, release engineering.

Success Criteria

  • Fresh machine → first issue through pipeline using only the docs (including description/attachment path).
  • Month of simulated use reconstructible from audit + artifacts (including original issue.md).
  • Load smoke test passes; backup → destroy → restore succeeds.
