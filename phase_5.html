Phase 5 Implementation Plan — Guardrails

📅 July 10, 2026 (revised) 📝 Status: Draft 🎯 Prerequisite: Phase 4 + project refactor + post-4 polish

Landed foundations — do not regress

Already in tree and recorded in spec.md (2026-07-10). Phase 5 work must not reintroduce free-text projects, title-only issues, current-phase-only drawers, or drop trigger/webhook bodies.

  • Projects — YAML registry only; budgets attach to registered projects.
  • Agent cast — named flavors frozen at submit.
  • Issue input — title + optional description + text-like attachments; dual-write SQLite + issue.md + attachments/.
  • Agent context — inline title/description; list attachment paths; previous phase output. Scope heuristics must use description, not title alone.
  • Artifact drawer — phase tabs; Result · Output/Workspace · Activity; Workspace tree + per-file diffs; no full Diff tab.
  • Workspace download — workspace.zip only when implementation is done.

If a Phase 5 design conflicts with the above, update the design — do not undo polish.

Goal

The system asks "are you sure?" before burning budget, halts runaway work mid-flight, and gives admins configurable escalation instead of silent failure.

Workstreams

A
Token Budget Enforcement
  • Budgets per agent, per issue, per YAML-registered project, per provider (spec §13.2).
  • Enforcement point: checked before each model call against events.jsonl usage — a hard mid-run stop.
  • On breach: phase → failed with budget_exceeded; admin notified; resumable after budget raise.
  • Warning threshold (e.g., 80%) → admin notification via the Phase 3 path.
  • Optional cost mapping for dashboard currency display; tokens remain the enforced unit.
B
Effort Estimation Gate
  • Planner emits effort tag via finish_task — not parsed from free-form output.*.
  • high inserts a human confirmation gate before implementation.
  • Threshold configurable per project.
C
Scope Detection
  • Heuristics over title + description + attachment basenames (not title alone).
  • Flagged issues require human confirmation before research begins.
  • Heuristics are advisory and logged.
D
Admin Escalation Rules
  • Configurable thresholds → escalations (failures, retries, budgets, adapter restarts, sandbox refusals).
  • Targets reuse Phase 3 notification adapters; rules in config + admin page.
E
MCP Per-Tool Granularity
  • Per-agent, per-tool allowlists (tightening Phase 4 per-server grants).
  • Tool-level constraints where supported; endpoint restrictions for HTTP-ish tools.
  • Dashboard surface for permissions — without replacing the multi-phase artifact drawer.

Tests

  • Budget hard stop mid-phase; resumable after raise.
  • Effort gate pauses pre-implementation.
  • Scope flag: pathological description (and title) held at submission.
  • Escalation: one notification, no alert storm.
  • MCP per-tool denial at call time.
  • Regression smoke: description + attachment submit; drawer on past phase; zip only when implementation done.

Success Criteria

A runaway task is stopped three ways: submission (scope), pre-implementation (effort), mid-run (budget). Guardrails compose with landed issue input and project registry.

Out of Scope

  • Metrics / deployment packaging → Phase 6
  • Changing multi-phase drawer / workspace tree / zip (already landed)
  • Project membership / invites (deferred)
