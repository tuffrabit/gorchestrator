Phase 5 Implementation Plan — Guardrails

📅 July 8, 2026 📝 Status: Draft 🎯 Prerequisite: Phase 4 complete

Goal

The system asks "are you sure?" before burning budget, halts runaway work mid-flight, and gives admins configurable escalation instead of silent failure. With external triggers live, unattended volume is now possible — this phase makes it safe.

Workstreams

A
Token Budget Enforcement
  • Budgets per agent, per issue, per project, per provider (spec §13.2).
  • Enforcement point: checked before each model call against accumulated per-call usage from events.jsonl — a hard mid-run stop, not a post-run report.
  • On breach: phase → failed with budget_exceeded; admin notified; issue paused, resumable after a human raises the budget.
  • Warning threshold (e.g., 80%) → admin notification via the Phase 3 path.
  • Optional cost mapping: maintainer-supplied per-model pricing table for dashboard currency display; tokens remain the enforced unit.
B
Effort Estimation Gate
  • Planner emits an effort tag (high/medium/low) as structured data via finish_task — not parsed out of free-form output.*, which the orchestrator never parses.
  • high inserts a human confirmation gate before implementation, regardless of configured adjudicator.
  • Threshold behavior configurable per project.
C
Scope Detection
  • Cheap heuristics at submission: text vagueness signals, forbidden-phrase list ("refactor the entire…"), planner file-count estimates.
  • Flagged issues require human confirmation before research begins — the cheapest point to stop a runaway task.
  • Heuristics are advisory and logged; false-positive rate reviewed via audit data.
D
Admin Escalation Rules
  • Configurable thresholds → escalations: consecutive failures, retry counts, budget warnings, adapter restart loops, sandbox refusals.
  • Targets reuse Phase 3 notification adapters; rules in config with an admin dashboard page.
E
MCP Per-Tool Granularity
  • Per-agent, per-tool allowlists (tightening Phase 4's per-server grants).
  • Tool-level constraints where supported (e.g., query_database restricted to SELECT); endpoint restrictions for HTTP-ish tools.
  • Dashboard surface showing exactly which agent can touch what — auditable at a glance.

Tests

  • Budget: scripted model with known usage → hard stop mid-phase; resumable after budget raise.
  • Effort gate: planner tags high → pipeline pauses pre-implementation; human confirm proceeds.
  • Scope flag: pathological issue text held at submission.
  • Escalation: N consecutive failures → one notification (no alert storm).
  • MCP: per-tool denial enforced at call time and absent from the agent's advertised toolset.

Success Criteria

A deliberately runaway task (huge scope, low budget) is stopped three separate ways: at submission (scope), before implementation (effort), and mid-run (budget). No guardrail can be bypassed by agent output content — all enforcement is orchestrator-side.

Out of Scope

  • Metrics/reporting polish, deployment packaging → Phase 6
