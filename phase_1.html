Phase 1 Implementation Plan — The Engine (CLI-Only)

📅 July 8, 2026 📝 Status: Complete — Frozen Record 🎯 Scope: Single Researcher run from CLI

Frozen historical record. Phase 1 shipped. Two items did not land as written: internal/trigger/ was never created (TriggerPort formalization deferred to Phase 4), and grep does not yet respect .gitignore (fixed in Phase 2 Project Cleanup). Superseded design details (custom LLMProviderPort, directory-scan adapter discovery) are recorded as revisions in spec.md.

Goal

Build a running system that accepts a manual trigger from the command line, runs a single Researcher agent, and writes the artifact triad to storage.

$ go run . run --issue="add auth" --project=foo
→ Creates issue, runs researcher, writes artifacts, marks done in SQLite

Design constraint: Everything is native Go. No CGO. The structure should assume the final product is a long-running app with a web interface and goroutine workers, but Phase 1 only exposes the CLI path.

Implementation Roadmap

1
Bootstrap

Initialize the module, root main.go, and a minimal stdlib flag CLI with placeholder run and version commands.

2
Configuration

Load global YAML from ~/.config/gorchestrator/config.yaml. Required fields:

  • storage_root — base directory for projects and workspaces
  • adapters_dir — directory to scan for external adapters
  • default_model — provider, model, API key env var, timeout
3
Storage Port

Define StoragePort and implement the OS filesystem adapter. Add artifact path helpers and constrain all paths under storage_root.

4
SQLite Schema

Open ~/.config/gorchestrator/gorchestrator.db. Create tables for projects, issues, and runs with repositories.

5
Adapter Foundation

Scan adapters_dir for manifests, parse YAML, implement JSON-RPC 2.0 over stdio, and manage child-process lifecycle. Include a tiny no-op test adapter.

6
LLM Provider Port

Build LLMProviderPort with OpenAI, local HTTP, and --dry-run stub adapters. The stub returns a canned markdown response so tests need no API key.

7
Core Tools

Implement native Go tools scoped to the Researcher: read_file, list_directory, grep_search, and write_output. The orchestrator resolves paths before they reach storage.

8
Researcher Agent

Define the Researcher struct, default system prompt, and tool bindings. Run it through ADK Go while keeping filesystem as the source of truth.

9
Orchestrator Run Command

Implement run: load/create project & issue, write task.json, spawn a goroutine, run n_loops, write result.json, and record the run in SQLite. Support clean context cancellation.

10
Integration Test

Run the orchestrator with --dry-run against a temp storage root. Assert artifacts exist and SQLite records are correct.

Filesystem Artifact Contract

Each Researcher run produces a single phase directory:

{storage_root}/projects/{project_id}/issues/{issue_id}/
└── research/
├── task.json ← orchestrator writes: agent config, model, loop mode, input context
├── output.md ← agent writes: free-form result, overwritten each loop
└── result.json ← orchestrator writes: status, tokens, duration, loop count

Success Criteria

  • go run . run --issue="add auth" --project=foo --dry-run completes without error.
  • Artifacts exist under the correct issue directory.
  • SQLite contains project, issue, and run records.
  • go test ./... passes.
  • No CGO dependencies are introduced.

Out of Scope

  • Planner and Implementer agents
  • self_done and human_gate handoff modes
  • Adjudicator port
  • Git workspace management
  • Web dashboard / HTTP server
  • MCP, SSO, notifications
  • Token budgets and guardrails
