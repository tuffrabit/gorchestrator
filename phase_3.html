Phase 3 Implementation Plan — Daemon, Dashboard & Auth

📅 July 8, 2026 📝 Status: Draft 🎯 Prerequisite: Phase 2 complete

Goal

$ gorchestrator serve

A team member logs in via OIDC, submits an issue from the dashboard (or CLI), watches the pipeline execute live, clicks retry with a reason at a human gate, and the agent re-runs with that feedback in context. Killing the daemon mid-run and restarting recovers every in-flight issue (spec §9.4).

Why daemonization is Workstream A, not a side effect: everything in this phase — webhook-style submission, real-time views, human gates that respawn goroutines in-process, parallel issues — presumes a persistent process with a work queue. The spec (§11.0) names this explicitly so it is scheduled, not discovered. The Phase 2 engine was built as an embeddable library precisely so this phase changes the front-end, not the core.

Workstreams

A
Daemonization
  • Engine API surface: SubmitIssue, Decide(issue, phase, decision, feedback), GetIssue, ListIssues, SubscribeEvents — CLI run/resume become thin callers.
  • serve subcommand: long-running process hosting engine, HTTP server, worker pool.
  • Issue queue + worker pool: SQLite-backed; configurable max_concurrent_issues; WAL/busy_timeout from cleanup are prerequisites.
  • Startup recovery scan: spec §9.4 state machine over all non-terminal issues; reconcile SQLite index against filesystem truth.
  • Human gates in-process: a decision respawns the worker goroutine directly; CLI resume remains for one-shot use.
  • Graceful shutdown: drain on SIGTERM; no failed statuses caused by shutdown — they are cancelled or recoverable in_progress.
B
HTTP API

POST /api/issues · GET /api/issues · GET /api/issues/{id} · GET /api/issues/{id}/artifacts/{path} · POST /api/issues/{id}/decisions · GET /api/events (SSE).

SSE per soft decision §17 Q4: per-issue streams tail events.jsonl + status transitions; polling fallback for degraded clients.

C
Dashboard (HTMX, server-rendered — §17 Q1)
  • Issue list: phase, status, attempt count, token burn — live via SSE.
  • Issue detail: artifact viewer (rendered Markdown, file tree, syntax highlighting), workspace diff view (snapshot vs. workspace), activity stream from events.jsonl, token display.
  • Adjudication UI: pass/fail/retry at any boundary regardless of configuration, with a first-class feedback field — warn when empty on retry/fail.
  • Notification center: pending gates, admin alerts on bad output.
  • No build pipeline: templates + HTMX attributes, static assets embedded in the binary.
D
Users, Auth, Audit
  • Migrations: users, sessions, audit_log; roles admin / member / viewer.
  • Authorization: viewer read-only; member adjudicate + submit; admin configuration + escalation targets.
  • OIDC (built-in): authorization-code flow, session cookie. SAML question (§17 Q6) decided at phase start.
  • Audit entries for every decision, submission, and configuration change (who, what, when, feedback).
E
Notifications (moved from Phase 2)
  • notifications table + dispatch loop in the daemon.
  • Built-in sink: console/log. External process adapters: Slack webhook and/or SMTP (§17 Q5 decided at phase start) — the first real JSON-RPC adapters, exercising the hardened client from cleanup.
  • Admin notifications on bad output (spec §11.4): errors, timeouts, empty outputs.

Tests

  • Daemon lifecycle: submit → pipeline completes → artifacts + SQLite consistent.
  • Kill -9 mid-phase → restart → recovery scan re-runs the phase (extends the Phase 2 test to daemon mode).
  • Human gate via API: decision with feedback → agent rerun includes feedback.
  • Concurrency: N parallel issues with a smaller worker pool; no SQLite lock errors.
  • AuthZ: viewer cannot decide; unauthenticated API rejected.
  • SSE: events observed during a dry-run pipeline.

Success Criteria

  • Team can log in via OIDC, watch a live run, retry with a reason, get a Slack/email alert on a human gate.
  • Daemon survives kill/restart with zero lost or corrupted issues.
  • CLI one-shot mode still works unchanged for local/headless use.

Out of Scope

  • MCP, external triggers (GitHub/Jira), git workspace, S3 → Phase 4
  • Token budgets, effort gates → Phase 5
