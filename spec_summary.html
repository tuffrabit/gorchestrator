AI Agent Orchestration System

Planning Session Summary — Architecture & Direction

📅 July 4, 2026 📝 Status: Planning Complete 🎯 Superseded — see spec.md (revised 2026-07-08)

What We're Building

A tight human + AI agent collaboration platform for software engineering teams. Not a generic "solve everything" orchestrator. Not an autonomous Devin clone. A structured, multi-agent pipeline that teams deploy against their own codebases and business problems.

The core insight: The human's job is casting — choosing which LLM plays which role. Deep research model for investigation. Fast planner for architecture. Competent coder for implementation. The system orchestrates the handoffs; the human configures the talent.

Core Principles

👤
Human-in-the-loop is first-class Humans observe, configure, and intervene. The system is transparent, not autonomous. A web dashboard is a first-class citizen, not an afterthought.
📁
Filesystem is the agent's memory Agents share data through filesystem artifacts, not shared memory or session state. This provides crash resilience, human readability, and freedom from schema lock-in.
⚡
Goroutines are a feature, not a bug In-process Go concurrency, SQLite, and filesystem storage are sufficient for hundreds of users. No Redis, no Postgres, no Kubernetes. If the process crashes, agents resume from filesystem state on restart.
🔌
Hexagonal architecture for extensibility Ports and adapters for storage, triggers, LLM providers, adjudication, and tools. Core stays small; extensions plug in cleanly. Adapters are external processes — no recompilation needed.
🔒
Secure by default Core tools are architecturally scoped by agent type. Agents do not get raw filesystem, shell, or git access. Security is the architecture, not a bolted-on permission layer.

The Pipeline

Trigger → Researcher → [Adjudicate] → Planner → [Adjudicate] → Implementer → Done

Each handoff boundary is configurable. Adjudication is optional and pluggable. The orchestrator manages the state machine; Google ADK Go powers the individual agent reasoning loops.

Key Architectural Decisions

Decision Choice Rationale
Go Language Go Concurrency model fits the agent pipeline perfectly. Goroutines are cheap. Context cancellation is native.
Core Agent Runtime Google ADK Go v2 Use for LLM provider abstraction and single-agent reasoning. Do NOT use for multi-agent orchestration or state management — we build that ourselves.
Decision State Storage SQLite + Filesystem SQLite for application state (projects, issues, users, audit log). Filesystem for agent artifacts and memory. No Postgres or Redis needed.
Decision Scale Target Hundreds of users In-process goroutines + SQLite + filesystem is sufficient. Horizontal scaling is explicitly out of scope for MVP.
Decision Agent Identity Baked-in Go structs Researcher, Planner, and Implementer are first-class citizens with default behavior. Configurable via YAML but not generic.
Decision Human Gates Goroutines die, respawn on decision No long-running sleep loops. Go concurrency is cheap. On restart, orchestrator reads SQLite and filesystem to resume.
Decision Config First YAML-backed, CLI-first Dashboard starts as observation + adjudication only. Strong YAML config enables future web-based configuration migration.
Decision Agent Output Completely free-form Content determined by system prompt + LLM. Orchestrator only enforces a minimal result.json status envelope. Next agent receives previous output as context.
Decision Adapter Model Built-in + External Process Common adapters compiled into core for zero-config. External JSON-RPC processes for exotic/enterprise needs. No recompilation required for extensions.
Decision Adapter Protocol JSON-RPC over stdio Simple, debuggable, language-agnostic, natural fit for MCP. Go stdlib JSON or goccy/go-json for serialization.
Decision WASM Deferred WASM via wazero is promising for high-frequency adapters, but over-complicates MVP. Revisit if external process overhead becomes measurable.
Security Shell Access None for core agents Core agents get typed tools, not raw shell. run_test is pre-configured and immutable. Security is architectural.
Security Git Access Orchestrator-managed only Agent never touches git. Orchestrator handles branch creation, workspace copy, commit, and push.
Security File Write Tool-scoped by agent type Researcher/Planner use write_output (orchestrator-resolved path). Implementer uses write_file (workspace-scoped). Different tools, not different permissions.

Adapter Architecture

Go is statically compiled. plugin.Open() is dead (Linux-only, fragile version matching). The core binary remains a single static executable. Extensibility is achieved through external process adapters communicating via JSON-RPC over stdin/stdout.

Two Adapter Categories

Category Description Examples
Built-in Compiled directly into the core binary. Zero external dependencies. Covers the 80% use case. OS filesystem storage, OpenAI/Anthropic LLM, webhook trigger, null/self adjudicator, OIDC auth, console logging
External Standalone binary discovered at runtime. Communicates via JSON-RPC over stdio. Language-agnostic. S3 storage, Jira trigger, email notification, SAML auth, custom LLM gateway

External Process Contract

Protocol: JSON-RPC 2.0 over stdin/stdout (line-delimited JSON).

Lifecycle:
1. Core discovers adapter binaries in ./adapters/ directory
2. Core spawns adapter as child process, holds stdin/stdout pipes open
3. Core sends initialize method with port interface schema
4. All calls are JSON-RPC request/response pairs
5. If adapter dies, core restarts with exponential backoff
6. For streaming (LLM tokens), adapter sends JSON-RPC notification messages

Adapter Manifest (YAML file next to binary):
name: s3
version: "1.0.0"
protocol: jsonrpc-stdio
port: storage
capabilities: [read, write, list, watch]

Example:
// Core → Adapter
{"jsonrpc":"2.0","method":"storage.read","params":{"path":"research/output.md"},"id":1}

// Adapter → Core
{"jsonrpc":"2.0","result":{"content":"...","exists":true,"size":1234},"id":1}

Port-Specific Adapter Strategy

Port Built-in External Rationale
Port StoragePort OS filesystem S3, Azure Blob High-frequency; external process overhead acceptable for MVP. Built-in filesystem covers 80%.
Port LLMProviderPort OpenAI, Anthropic, Gemini, local HTTP Enterprise gateways, custom APIs Common providers built-in for zero-config. Exotic providers use external process.
Port TriggerPort Manual CLI, HTTP webhook GitHub Issues, Jira, email, Slack bot Triggers are naturally long-running listeners. External process is a good fit.
Port AdjudicatorPort Null, Self, Human Custom AI reviewer Core adjudication modes built-in. Custom reviewers can be external.
Port MCPAdapterPort — Any MCP server MCP is natively an external process protocol. Core implements MCP client.
Port NotificationAdapterPort Console logging Email (SMTP), Slack webhook Low-frequency, naturally external.
Port SSOAdapterPort OIDC SAML OIDC is standard HTTP flow, built-in. SAML is complex enough for external.

Future: WebAssembly Adapters

WASM via wazero is a candidate for future phases if external process overhead becomes a bottleneck. This would introduce a third adapter category:

  • WASM adapters: Sandboxed, near-native performance, single-binary deployment (core + .wasm files)
  • Use case: High-frequency ports like StoragePort where process spawn overhead is measurable

Deferred to Phase 4 or beyond. Not in MVP scope.

Security Model: Secure by Default

Security is not a permissions layer bolted on top — it is the architecture itself. Agents do not get raw filesystem, shell, or git access. They receive orchestrator-mediated tools that are scoped by design. Each agent type receives a different tool binding, not the same tools with different permissions.

Core Tool Matrix

Tool Researcher Planner Implementer Description
read_file ✅ ✅ ✅ Read file content (full or partial). Paths resolved against orchestrator allowlist.
list_directory ✅ ✅ ✅ List contents of a directory.
grep_search ✅ ✅ ✅ Pattern search across files.
write_output ✅ ✅ ❌ Write to the agent's designated output.* file. Orchestrator resolves path; agent doesn't know the filesystem layout.
write_file ❌ ❌ ✅ Write a file within the implementer's workspace. Path is workspace-relative.
update_file ❌ ❌ ✅ Update/patch a file within the implementer's workspace.
run_test ❌ ❌ ✅ Execute the project's pre-configured test command in a sandboxed subprocess with timeout.
bash / shell_exec ❌ ❌ ❌ Not available to any core agent.

Key Principles

  • Researcher and Planner never write files directly. They use write_output, which is an orchestrator-provided tool that only writes to the current agent's output.* path in the issue directory. The agent does not know the path; the orchestrator resolves it. This is not a "permission" — it is a completely different tool with a single hardcoded destination.
  • Implementer writes to a workspace. The write_file and update_file tools are bound to the implementer's workspace directory. The agent sees relative paths (e.g., src/auth.go) but the orchestrator resolves them to the workspace absolute path.
  • No shell access. Core agents do not have bash or shell_exec tools. All operations are mediated through typed tool calls.
  • Path allowlist. The orchestrator maintains an allowlist of readable paths for each agent run. Attempts to access paths outside the allowlist are rejected by the orchestrator before reaching the StoragePort.

MCP Permission Model

MCP servers are the real permission surface because they expose arbitrary capabilities (database access, API calls, internal service invocations).

MVP Model:

  • MCP servers are project-level configuration
  • All tools from an enabled MCP server are available to all agents
  • The human maintainer decides which MCP servers to connect
  • Default: deny — no MCP servers are connected unless explicitly configured

Future (Phase 4+):

  • Per-agent, per-tool allowlists
  • Tool-level permission granularity (e.g., query_database only allows SELECT)
  • Endpoint restrictions for API tools

Git Workspace Model

The core implementer agent and toolset are geared around a targeted, pre-existing git repository. No agent can create a new git repository or GitHub/Bitbucket project via core tools. Repository management is a human responsibility.

Prerequisites

  • git must be installed and configured in the environment
  • gh CLI is optional but recommended for GitHub-specific workflows
  • SSH keys, tokens, or gh profiles must be configured by the human maintainer outside the application
  • The application does not manage auth setup; it assumes the environment is ready

Project-Level Git Configuration

project.yaml
project:
  name: "auth-service"
  git:
    repo_url: "git@github.com:myorg/auth-service.git"
    base_branch: "develop"
    auth:
      type: "ssh_key"  # or "token", "gh_cli"
      ssh_key_path: "/secrets/deploy_key"
      # or:
      # type: "gh_cli"
      # profile: "work"

Implementer Workspace Lifecycle

The agent never touches git. The orchestrator handles the entire git lifecycle:

  1. Branch Creation: Orchestrator creates a unique branch per implementer run:
    git checkout -b ai-implementer/{issue_id}-{timestamp}
  2. Workspace Copy: Orchestrator copies repo files into the configured storage location:
    workspaces/{issue_id}/implementer-{run_id}/
  3. Agent Execution: Implementer receives read_file, write_file, update_file tools scoped to that workspace. The agent sees relative paths; the orchestrator resolves to absolute.
  4. Post-Execution: Orchestrator stages changes, commits with a structured message, and optionally pushes. The human decides whether to merge.

Parallel Implementers: Each run gets its own branch and workspace. No collision. The orchestrator tracks workspace_id and branch_name in SQLite.

Test Execution (run_test Tool)

The implementer has access to a run_test core tool that executes the project's pre-configured test command in a sandboxed subprocess.

project.yaml
project:
  test_command: "go test ./..."
  test_timeout: "60s"
  # Or more complex:
  # test_command: "docker run --rm -v $(pwd):/app myproject-test-runner"
  # test_command: "./scripts/run-integration-tests.sh"
  # test_command: "python -m pytest tests/"

Properties:

  • Command is immutable — defined in project config, not modifiable by the agent
  • Executed in the implementer's workspace directory
  • Sandboxed subprocess with timeout and output capture
  • Agent receives stdout/stderr but cannot modify the command
  • Enables implementer test-and-fix loops without opening arbitrary shell access

Filesystem Artifact Contract

Each issue gets a directory tree. The orchestrator manages a minimal status envelope; agents write whatever they want.

projects/{project_id}/issues/{issue_id}/
├── research/
│ ├── task.json ← Orchestrator: instructions, model config, loop config
│ ├── result.json ← Orchestrator: status, error, tokens, done rationale
│ └── output.* ← Agent: COMPLETELY FREE-FORM (.md, .py, .json, etc.)
├── plan/
│ ├── task.json
│ ├── result.json
│ └── output.*
├── adjudication/
│ ├── task.json
│ ├── result.json
│ └── output.*
└── implementation/
├── task.json
├── result.json
└── output.*

workspaces/{issue_id}/
└── implementer-{run_id}/ ← Git workspace copy for implementer

Critical rule: No agent reads or parses result.json. Only the orchestrator manages status. Agents read their task.json and write their output.*.

Agent Handoff Modes

Each handoff boundary is independently configurable:

Mode Behavior Use Case
n_loops Agent runs exactly N times. Orchestrator tracks count. After N loops, status becomes done regardless of output quality. "Don't overthink it" agents
self_done Agent evaluates itself against an English rubric. Writes assessment to output.*. Orchestrator reads result.json status. Agents that can judge their own completeness
human_gate Agent completes, status becomes waiting_human. Goroutine dies. Human reviews via dashboard and clicks pass/fail/retry. New goroutine spawned on decision. Critical phases requiring human judgment

The Engine CLI-Only

A running system that accepts a trigger, runs a no-op agent, and writes artifacts to storage.

  • Trigger: manual CLI adapter (built-in)

  • n_loops handoff mode only

  • Adapter discovery framework: scan ./adapters/, load manifests, spawn external processes
  • JSON-RPC stdio protocol foundation

  • Core toolset: read_file, list_directory, grep_search, write_output, write_file, update_file
  • Handoff modes: n_loops, self_done, human_gate
  • Adjudication port: NullAdjudicator, SelfAdjudicator (built-in)

What "Bad Output" Means

For the MVP, "bad output" triggers admin notifications:

  • LLM API errors or timeouts
  • Agent exceptions during tool execution
  • Configuration errors or missing required config
  • Empty output files (zero-byte output.* heuristic)

Heuristic-based "badness" detection (confidence scores, quality analysis) is deferred to Phase 5. The adjudicator port supports null, self, and human adapters. A dedicated AI adjudicator is a future possibility.
